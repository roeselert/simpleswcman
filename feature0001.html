US-01 – Geschäftsprozesse & Informationen erfassen
Strukturierte Aufnahme aller relevanten Geschäftsprozesse inkl. der verarbeiteten Informationen.
US-01
Geschäftsprozesse dokumentieren
Informationsverbund – Kernprozessaufnahme
Strukturanalyse Geschäftsprozesse CRUD Pflicht
Als Informationssicherheitsbeauftragter (ISB) möchte ich Geschäftsprozesse meiner Organisation mit allen relevanten Informationen, Anwendungen und IT-Systemen strukturiert erfassen und pflegen, damit ich die Grundlage für die Schutzbedarfsfeststellung und das Sicherheitskonzept habe.
Datenmodell
``` GESCHAEFTSPROZESS {
  id UUID [PK, required]
  bezeichnung String(200) [required]
  beschreibung Text [optional]
  verantwortlicher FK → BENUTZER [required]
  organisationseinheit String(100) [required]
  kritikalitaet Enum(NORMAL|WICHTIG|KRITISCH) [required]
  erstellt_am DateTime [auto]
  geaendert_am DateTime [auto]
}

INFORMATION {
  id UUID [PK]
  bezeichnung String(200) [required]
  kategorie Enum(PERS_DATEN|FINANZDATEN|KNOW_HOW|BETRIEBSDATEN|SONSTIGE)
  schutzbedarf_C Enum(NORMAL|HOCH|SEHR_HOCH) // Vertraulichkeit
  schutzbedarf_I Enum(NORMAL|HOCH|SEHR_HOCH) // Integrität
  schutzbedarf_A Enum(NORMAL|HOCH|SEHR_HOCH) // Verfügbarkeit
  prozess_id FK → GESCHAEFTSPROZESS
}

// M:N Verknüpfung: Prozess ↔ Anwendung ↔ IT-System
PROZESS_ANWENDUNG { prozess_id, anwendung_id }
PROZESS_ITSYSTEM { prozess_id, itsystem_id }
```
Prozessbeschreibung
1
Erstaufnahme importieren / anlegen
ISB legt neuen Prozess an oder importiert aus Erstaufnahme (CSV/Excel). Pflichtfelder: Bezeichnung, Verantwortlicher, Org.-Einheit, Kritikalität.
2
Informationen zuordnen
Pro Prozess werden verarbeitete Informationen erfasst (z. B. Kundendaten, Produktionspläne). Schutzbedarf (C/I/A) wird je Information bewertet.
3
Anwendungen & IT-Systeme verknüpfen
Verknüpfung mit bereits erfassten Anwendungen (US-04) und IT-Systemen (US-03) via Many-to-Many Relation.
4
Review & Freigabe
Prozessverantwortlicher bestätigt Richtigkeit. ISB oder CISO gibt Datensatz frei. Änderungen werden auditiert (Änderungshistorie).
5
Export & Berichterstattung
Prozessliste kann als BSI-konformes PDF oder Excel exportiert werden. Wird automatisch in Sicherheitskonzept übernommen.

US-02 – Netzplan erstellen & verwalten
Digitaler Netzplan mit topologischer Darstellung aller Netzsegmente, Verbindungen und Zonen.
US-02
Netzplan dokumentieren
Infrastruktur – Netzwerkarchitektur
Netzplan Infrastruktur Topologie Sicherheitszonen
Als IT-Administrator möchte ich den Netzplan des Informationsverbundes mit allen Segmenten, aktiven Netzkomponenten, Verbindungen und Sicherheitszonen erfassen und visualisieren, damit Sicherheitsverantwortliche die Netzstruktur verstehen und Maßnahmen ableiten können.
Datenmodell
``` NETZSEGMENT {
  id UUID [PK]
  bezeichnung String(100) [required] // z.B. “DMZ”, “Büronetz”
  netzadresse CIDRField [optional] // z.B. 192.168.10.0/24
  vlan_id Integer [optional]
  zone Enum(INTERN|DMZ|EXTERN|OT|MANAGEMENT)
  schutzbedarf Enum(NORMAL|HOCH|SEHR_HOCH)
  standort_id FK → LIEGENSCHAFT
}

NETZKOMPONENTE {
  id UUID
  bezeichnung String(100) [required]
  typ Enum(ROUTER|SWITCH|FIREWALL|WAP|PROXY|VPN_GW)
  hersteller String(100)
  modell String(100)
  ip_adresse InetField
  segment_id FK → NETZSEGMENT
  itsystem_id FK → ITSYSTEM [optional]
}

NETZVERBINDUNG {
  id UUID
  von_segment FK → NETZSEGMENT
  zu_segment FK → NETZSEGMENT
  protokoll String(50) // TCP/UDP/IPSec etc.
  verschluesselt Boolean [required]
  firewall_regel Text [optional]
}
```
Prozessbeschreibung
1
Netzsegmente definieren
IT-Admin legt alle Netzsegmente mit CIDR-Block, VLAN-ID und Sicherheitszone an. Verknüpfung mit Liegenschaft/Gebäude (US-05).
2
Netzkomponenten erfassen
Router, Switches, Firewalls, WLAN-AP werden je Segment zugeordnet. Optional: Import aus CMDB oder SNMP-Discovery.
3
Verbindungen & Übergänge dokumentieren
Segmentübergänge inkl. Protokoll, Verschlüsselung und Firewall-Regelreferenz werden eingetragen. Systemgrenzen zur externen Kommunikation markieren.
4
Topologie-Visualisierung
Automatische Darstellung als interaktiver Netzplan (Graph). Export als SVG/PDF für das Sicherheitskonzept.

Akzeptanztests (Given-When-Then)
🧪 AT-02.1 – CIDR-Validierung
GIVENIT-Admin legt neues Netzsegment an
WHENEr als Netzadresse „192.168.999.0/24" eingibt (ungültige IP)
THENSystem zeigt „Ungültige CIDR-Notation" und verweigert das Speichern
🧪 AT-02.2 – Segmentübergang ohne Verschlüsselung
GIVENZwei Segmente „Büronetz (INTERN)" und „DMZ" existieren
WHENIT-Admin eine Verbindung ohne Verschlüsselung anlegt
THENZeigt das System eine Sicherheitswarnung „Unverschlüsselter Übergang zu Zone DMZ – Bitte prüfen"
ANDDie Verbindung kann nach expliziter Bestätigung trotzdem gespeichert werden
🧪 AT-02.3 – Netzplan Export
GIVENMindestens 3 Segmente und 5 Komponenten sind erfasst
WHENISB auf „Netzplan als PDF exportieren" klickt
THENWird ein PDF mit der grafischen Topologie und einer tabellarischen Komponentenliste generiert
ANDDas Dokument trägt Datum, Versionsnummer und Erstellername

US-03 – IT-Systeme inventarisieren
Vollständige Inventarisierung aller IT-Systeme inkl. Typ, Betriebssystem, Standort und Schutzbedarf.
US-03
IT-Systeme erfassen & klassifizieren
Server, Clients, Mobile Devices, IoT, OT
IT-Inventar Systemkatalog Kritische Systeme OT/IoT
Als IT-Administrator möchte ich alle im Informationsverbund betriebenen oder geplanten IT-Systeme mit technischen Details, Standort, Schutzbedarf und Verantwortlichkeiten erfassen, damit der ISB darauf aufbauend Gefährdungen und Maßnahmen identifizieren kann.
Datenmodell
``` ITSYSTEM {
  id UUID [PK]
  inventarnummer String(50) [required, unique]
  bezeichnung String(200) [required]
  typ Enum(SERVER|CLIENT|NETZWERK|MOBILE|DRUCKER|OT|IOT|SONSTIGE)
  plattform Enum(PHYSISCH|VM|CONTAINER|CLOUD_SAAS|CLOUD_IAAS)
  betriebssystem String(100) [optional]
  os_version String(50)
  hostname String(100)
  ip_adressen JSON[] // mehrere IPs möglich
  mac_adresse MACField [optional]
  status Enum(PRODUKTIV|TEST|GEPLANT|ABGEKUENDIGT)
  schutzbedarf Enum(NORMAL|HOCH|SEHR_HOCH) // abgeleitet via Vererbung
  verantwortlicher FK → BENUTZER
  raum_id FK → RAUM [optional]
  segment_id FK → NETZSEGMENT
  anschaffungsdatumDate [optional]
  end_of_support Date [optional]
  bemerkung Text
}
```
Prozessbeschreibung
1
Systeme anlegen / importieren
Manuelle Erfassung oder Import via CSV/CMDB-Schnittstelle. Inventarnummer wird automatisch generiert wenn nicht vorhanden.
2
Klassifizierung & Plattform
Typ (Server, Client, OT…) und Plattform (physisch, VM, Cloud) werden festgelegt. EoS-Datum löst Warnhinweise aus.
3
Standort & Netz zuordnen
Verknüpfung mit Raum (US-05) und Netzsegment (US-02). Physischer Ort für mobile Systeme als Hinweis.
4
Schutzbedarf vererben
Der Schutzbedarf des Systems wird aus den darauf laufenden Anwendungen (US-04) abgeleitet (Maximumprinzip), kann aber manuell überschrieben werden.

US-04 – Anwendungen & Abhängigkeiten dokumentieren
Katalog aller eingesetzten Softwareanwendungen mit Abhängigkeiten zu Prozessen und IT-Systemen.
US-04
Anwendungskatalog pflegen
Software – Abhängigkeiten – Datenhaltung
Anwendungen Abhängigkeiten Softwarekatalog
Als Informationssicherheitsbeauftragter möchte ich alle eingesetzten Anwendungen mit ihren Abhängigkeiten zu Geschäftsprozessen und IT-Systemen sowie den darin verarbeiteten Daten dokumentieren, damit ich Risiken aus Softwareveralterung oder Abhängigkeitsbrüchen frühzeitig erkennen kann.
Datenmodell
``` ANWENDUNG {
  id UUID [PK]
  bezeichnung String(200) [required]
  hersteller String(100)
  version String(50)
  lizenzmodell Enum(KAUF|MIETE|OPEN_SOURCE|EIGENENTWICKLUNG)
  betriebsart Enum(ON_PREMISE|CLOUD|HYBRID)
  datenhaltung_ort String(200) // Serverstandort / Cloudregion
  personendaten Boolean // DSGVO-relevant?
  schnittstellen JSON[] // API/Schnittstellen zu anderen Anwendungen
  verantwortlicher FK → BENUTZER
  status Enum(PRODUKTIV|TEST|GEPLANT|ABGEKUENDIGT)
}

// Anwendung läuft auf IT-System (M:N)
ANWENDUNG_ITSYSTEM {
  anwendung_id FK itsystem_id FK rolle String
}
```
Prozessbeschreibung
1
Anwendung anlegen
ISB oder IT-Admin erfasst Anwendungsname, Hersteller, Version, Lizenzmodell und Betriebsart.
2
Datenhaltung & DSGVO
Ort der Datenhaltung (On-Premise/Cloud-Region) und Personenbezug werden dokumentiert. DSGVO-Flag aktiviert zusätzliche Pflichtfelder.
3
IT-Systeme verknüpfen
Anwendung wird den IT-Systemen (US-03) zugeordnet auf denen sie läuft. Rolle (Appserver, DB, Client) wird angegeben.
4
Schnittstellen erfassen
APIs und Verbindungen zu anderen Anwendungen werden dokumentiert. Abhängigkeitsmatrix wird automatisch generiert.

US-05 – Liegenschaften & Räume erfassen
Räumliche Infrastruktur: Standorte, Gebäude, Serverräume, Produktionshallen.
US-05
Räumliche Gegebenheiten dokumentieren
Liegenschaften – Gebäude – Räume – Zutritt
Liegenschaften Gebäude Zutrittskontrolle Produktion
Als Facility-Manager / IT-Admin möchte ich alle Liegenschaften, Gebäude und relevanten Räume (insbesondere Serverräume und Produktionsstätten) mit ihren physischen Sicherheitsmerkmalen erfassen, damit Zutrittskontrolle und physische Schutzmaßnahmen im Sicherheitskonzept berücksichtigt werden.
Datenmodell
``` LIEGENSCHAFT {
  id UUID
  bezeichnung String(200) [required]
  adresse String(300)
  land ISO3166
}

GEBAEUDE {
  id UUID
  bezeichnung String(100) [required]
  liegenschaft_id FK → LIEGENSCHAFT
  stockwerke Integer
  baujahr Year
}

RAUM {
  id UUID
  bezeichnung String(100) [required]
  raumnummer String(20)
  gebaeude_id FK → GEBAEUDE
  typ Enum(BUERO|SERVERRAUM|RZ|PRODUKTION|LAGER|BESPRECHUNG|SONSTIGE)
  etage Integer
  zutrittskontrolle Enum(KEINS|SCHLUESSEL|TRANSPONDER|BIOMETRIE)

  einbruchschutz Enum(KEINS|BASIS|ERHOEHEN|HOCH)
  klimatisierungBoolean

Prozessbeschreibung
1
Liegenschaft anlegen

Gebäudestruktur erfassen
Je Liegenschaft werden Gebäude mit Stockwerkanzahl und Baujahr erfasst.
3
Räume & physische Sicherheit
Räume werden Gebäuden zugeordnet. Für Serverräume/RZ werden Pflichtfelder Zutrittskontrolle, Brandschutz, Klimatisierung und USV ausgefüllt.
4
IT-Systeme verorten
IT-Systeme (US-03) werden ihrem physischen Raum zugewiesen. Visualisierung als Gebäudeplan optional.

US-06 – Schutzbedarf feststellen & berichten
Konsolidierte Schutzbedarfsfeststellung und Bericht für das Management.
US-06
Schutzbedarf konsolidieren & Bericht erstellen
Sicherheitskonzept – Management-Report
Kritisch Management BSI-Konform Export
Als CISO / Geschäftsführung möchte ich eine konsolidierte Schutzbedarfsfeststellung mit einer Übersicht aller kritischen Systeme, Anwendungen und Prozesse sowie einen druckfertigen BSI-konformen Bericht erhalten, damit ich fundierte Entscheidungen über Sicherheitsinvestitionen treffen kann.
Datenmodell
``` SCHUTZBEDARF_BERICHT {
  id UUID
  titel String(200)
  version String(20)
  erstellt_von FK → BENUTZER
  erstellt_am DateTime
  freigegeben_von FK → BENUTZER [nullable]
  status Enum(ENTWURF|REVIEW|FREIGEGEBEN)
  zusammenfassung Text
}

SCHUTZBEDARF_POSITION {
  bericht_id FK → SCHUTZBEDARF_BERICHT
  objekt_typ Enum(PROZESS|ANWENDUNG|ITSYSTEM)
  objekt_id UUID
  schutzbedarf_C Enum(NORMAL|HOCH|SEHR_HOCH)
  schutzbedarf_I Enum(NORMAL|HOCH|SEHR_HOCH)
  schutzbedarf_A Enum(NORMAL|HOCH|SEHR_HOCH)
  begruendung Text
}
```
Prozessbeschreibung
1
Schutzbedarf automatisch konsolidieren
System aggregiert Schutzbedarf aller Objekte. Vererbung: Informationen → Prozesse → Anwendungen → IT-Systeme (Maximumprinzip).
2
ISB prüft & begründet
ISB kontrolliert abgeleitete Schutzbedarfe, fügt Begründungen hinzu und kann Werte begründet anpassen.
3
CISO Review & Freigabe
CISO erhält Benachrichtigung, prüft den Bericht und gibt ihn frei oder sendet ihn zur Überarbeitung zurück.
4
BSI-Bericht exportieren
Freigegebener Bericht wird als PDF nach BSI IT-Grundschutz Struktur exportiert. Automatische Versionierung.

``` // ═══════════════════════════════════════════════════════
// KERN-ENTITÄTEN des BSI-Grundschutz Informationsverbunds
// ═══════════════════════════════════════════════════════

BENUTZER ──── verwaltet ────► GESCHAEFTSPROZESS
                            │
                            ├── 1:N ──► INFORMATION (C/I/A Schutzbedarf)
                            ├── M:N ──► ANWENDUNG
                            └── M:N ──► ITSYSTEM

ANWENDUNG ─────── läuft auf ──► ITSYSTEM (M:N via Rolle)
                      └── verbunden mit ─► ANWENDUNG (Schnittstellen M:N)

ITSYSTEM ──────── platziert in ─► RAUM
                  └── verbunden über ─► NETZSEGMENT

NETZSEGMENT ───── enthält ──────► NETZKOMPONENTE
                 └── verbunden mit ─► NETZSEGMENT via NETZVERBINDUNG
                 └── liegt in ──────► LIEGENSCHAFT

LIEGENSCHAFT ──── hat 1:N ──────► GEBAEUDE
GEBAEUDE ────────── hat 1:N ──────► RAUM

SCHUTZBEDARF_BERICHT ── aggregiert ─► alle Entitäten (CIA)
                        └── Vererbung: INFORMATION → PROZESS → ANWENDUNG → ITSYSTEM

// Querschnittsthemen
AUDIT_LOG ── protokolliert alle CREATE/UPDATE/DELETE + Auth-Events
BENUTZER_ROLLE ── M:N (Benutzer kann mehrere Rollen haben, je Verbund)
```
